# AWS Messaging & Microservices Notes

## Services Overview

### SES (Simple Email Service)
- Used for sending and receiving emails at scale.

### SQS (Simple Queue Service)
- Fully managed message queue for decoupling services.
- Workflow:
  1. Create a queue
  2. Producers send messages to the queue
  3. Consumers subscribe/poll the queue to read messages

### SNS (Simple Notification Service)
- Pub/Sub messaging service.
- Sends messages to multiple subscribers (e.g., SQS, email, Lambda).

## Microservices Architecture

- Breaking an application into smaller, independent services is called **microservices architecture**.

### Common Pitfall: Shared Databases
- Multiple microservices sharing the same database is **not recommended**.
- Issues:
  - Tight coupling between services
  - Risk of **inconsistent reads**
  - Example:
    - Service A writes data
    - Service B reads it immediately → may get stale or partial data

### Recommended Approach
- Each microservice should have:
  - Its **own database**
  - Clear ownership of its data

- Instead of direct database access:
  - Services communicate via **APIs** or **events**
  - Each service is responsible for managing and exposing its own data

### Benefits
- Better isolation
- Improved scalability
- Easier maintenance
- Reduced risk of data inconsistency
